### 検索範囲を絞る3つのレイヤー

| レイヤー | 誰が | どこで | 何をする |
|---------|------|--------|---------|
| **レイヤー1：事前選択** | 管理者 | Hikari管理画面 | 各データソースの書類/フォルダにチェックをつけて検索対象を設定 |
| **レイヤー2：データソース選択** | エンドユーザー | Direct UI | 5種のデータソースボタンで検索範囲を絞る |
| **レイヤー3：AI自動絞込み** | AI | バックエンド | チェック済み×選択データソースの中から、質問に関連する書類を自動特定 |

```
レイヤー1              レイヤー2              レイヤー3
管理者が事前に          ユーザーが              AIが質問に応じて
チェック（Hikari）      データソース選択         自動で絞込み
                      （Direct）

全書類 600件+    →    チェック済み     →    選択DB内の     →    関連書類
                      書類のみ              チェック済み          5〜10件
                      (例: 200件)           (例: 100件)          → RAG検索
```

### データソースとフェーズの対応

| データソース | 内容 | 検索対象の単位 | 対応フェーズ |
|-------------|------|--------------|------------|

## 1. エンドユーザーUI — Direct上のチャット体験

### 1-1. 現状の課題

- テキストで質問を打つだけ → **何を聞けるか分からない**（初見ユーザーの壁）
- 回答が長文になりがち → **1,024文字制限**で分割送信 → 読みにくい
- 根拠資料の参照が困難 → 「○○規定 第○条」とだけ表示 → **原文を見たい**
- 複数データソースの存在を意識しにくい → **今どのDBを検索しているか不明**
- **建設DBは書類数が多く、共通文書はフォルダ数が多い** → 全件検索すると回答速度が大幅に低下する

### 1-2. 核心の設計課題：検索範囲と回答速度のトレードオフ

#### 問題の構造

```
全件検索（50書類 + 共通文書全フォルダ）
  → 精度は高いが、回答に 15〜30秒以上かかる
  → 現場で待てない。使われなくなる。

絞り込み検索（関連書類のみ）
  → 回答は 3〜5秒で返せる
  → ただし、ユーザーが「どの書類を検索すべきか」を知らない場合がある
```

**この問題をUIでどう解くか**が、弘電社PJのUI/UX設計における最大の論点。

#### 前提：データの規模感

```
建設データベース
  └─ 300種類以上の書類（内線規定、電気設備基準、高圧受電規定...）
  └─ 今後も増え続ける

共通文書
  └─ 300以上のフォルダ（施工標準、安全管理、品質管理...）
  └─ 今後も増え続ける

→ 合計600以上の検索対象。ユーザーに選ばせるUIは非現実的。
→ 「本棚から選ぶ」モデルは300冊の本棚を見せることになり破綻する。
```

**この規模では、UIで解決するのではなく、検索の仕組み（バックエンド）で解く必要がある。**
UIの役割は「ユーザーに検索先を選ばせること」ではなく、
**「AIが何を検索したかを透明に見せ、足りなければ拡張できるようにすること」**に変わる。

#### 提案：3層の検索アーキテクチャ

---

#### 第1層：AI自動ルーティング（バックエンド）

ユーザーは質問するだけ。バックエンドで検索範囲を自動決定する。

```
処理フロー：

ユーザー：「ケーブルラックの施工方法を教えて」
    │
    ▼
[Step 1] インデックス検索（0.3〜0.5秒）
    │
    │  600以上の書類/フォルダの「メタ情報インデックス」を検索
    │  （書類名・目次・章タイトル・キーワードの軽量インデックス）
    │  → 関連度の高い 5〜10件に絞り込み
    │
    │  ※ 書類の本文全体を検索するのではなく、
    │    「どの書類が関連しそうか」だけを高速に判定する
    │
    ▼
[Step 2] 本文検索（RAG 2〜4秒）
    │  → 絞り込んだ 5〜10件の書類のみ本文を検索
    │
    ▼
[Step 3] 回答生成 + 検索範囲の明示
```

**メタ情報インデックスのイメージ**:

```
書類ID | 書類名                    | カテゴリ      | キーワード
───────┼──────────────────────────┼─────────────┼──────────────────
001    | 内線規定 2022年版         | 配線・施工    | 許容電流, 絶縁, 接地...
002    | 電気設備技術基準           | 技術基準      | 絶縁耐力, 安全距離...
003    | 高圧受電設備規定           | 受電設備      | 高圧, 受電, 試験電圧...
...    | ...                      | ...          | ...
301    | 施工標準（ケーブルラック）  | 社内/施工     | ケーブルラック, 支持間隔...
...
```

この「目次レベル」のインデックスなら600件でも0.5秒以内で検索可能。
書類の追加時にメタ情報を登録するだけなので、増え続けても対応できる。

**ポイント**:
- 「どの書類に書いてあるか」はAIが判断するので、ユーザーは知らなくてOK
- 書類が300→500→1000と増えてもUIは一切変わらない
- インデックスの精度がシステム全体の品質を決めるため、ここに投資する価値がある

#### 第3層：ユーザーの検索パターン学習（中長期）

利用ログを蓄積し、システムが賢くなっていく仕組み。

```
学習サイクル：

利用ログの蓄積
    │
    ▼
パターンの発見
    │  ・「ケーブルラック」を含む質問 → 90%の確率で内線規定+施工標準を参照
    │  ・工事部のユーザー → 施工標準の利用率が高い
    │  ・営業部のユーザー → 技術基準の利用率が高い
    │
    ▼
ルーティング精度の向上
    │  ・質問パターン別の検索先マッピングを更新
    │  ・部門別のデフォルト検索範囲を最適化
    │
    ▼
管理画面でのフィードバック
    ・「この質問でこの書類が選ばれなかった」→ ルーティングルールに反映
    ・利用頻度の低い書類カテゴリの見直し提案
```

**フェーズ3のユーザー権限管理と連動**:
- 部門ごとに「デフォルト検索範囲」を設定可能にする
- 工事部 → 施工関連の書類を優先的に検索
- 営業部 → 技術基準・規定関連を優先的に検索
- 全社共通の書類は常に検索対象に含める

---

#### 確定方針：入口は1つ、シンプルに

```
ユーザーの意図は2パターンあるが、入口は1つで両方対応する：

パターン1：「○○ってどうだったっけ？」
  → データソースボタンで絞り（または「すべて」のまま）、質問するだけ

パターン2：「あの書類のあれを調べたい」
  → 質問文に書類名を含める（例：「内線規定で ○○」）
  → AIがその書類を優先して検索
```

入口を分けると迷う。**質問を打つだけ**のシンプルなUIに統一する。

- デフォルトは「すべて」→ 迷わず使い始められる
- 特定のデータソースに絞ると速度が上がる → ユーザーが体感で学習する
- データソースを絞った場合でも、「他のDBにも情報があるかも」とサジェストする
- 質問文に書類名を含めると、その書類を優先的に検索（自然言語ルーティング）
  - 例：「**内線規定で**ケーブルラックの支持間隔」→ 内線規定を最優先
  - 例：「ケーブルラックの支持間隔」→ AIが自動判定
- 回答に「検索した書類」を必ず明示 → ユーザーが見落としに気づける
- 新データソース追加時はフィルタボタンを1つ追加するだけ（例：📐 Autodesk）

---

#### 特定の書類を検索したい場合

入口を分けず、**質問文に書類名を含める**ことで対応する。

```
パターン1（書類を指定しない）:
  ユーザー：「ケーブルラックの支持間隔を教えて」
  → AIが全チェック済み書類から自動で関連書類を特定

パターン2（書類を指定する）:
  ユーザー：「内線規定で ケーブルラックの支持間隔を教えて」
  → 内線規定を優先的に検索。他の関連書類も補足的に参照

パターン3（データソース + 書類指定）:
  ユーザー：[📋 建設DB] を選択
  ユーザー：「内線規定で ケーブルラックの支持間隔を教えて」
  → 建設DB内の内線規定に絞って高速検索（2〜3秒）
```

**ポイント**:
- 書類名を含めなくてもAIが自動で探す → 初心者でも使える
- 書類名を含めれば優先的に検索 → ベテランは意図通りの結果を得られる
- UIの入口は1つのままで、質問の書き方で自然にコントロールできる

---

#### バックエンドとの対応

| UI操作 | バックエンドの処理 |
|--------|------------------|
| 「すべて」で質問 | メタインデックス検索(0.5秒) → 書類絞込み → RAG(2〜4秒) |
| データソースボタンで絞って質問 | 選択DB内のチェック済み書類から絞込み → RAG(2〜4秒) |
| 質問文に書類名を含める | 該当書類を優先的にルーティング → RAG(2〜3秒) |
| 🔍 もっと広く検索（回答後） | チェック済み全書類でRAG(15〜30秒) |

  300件 → 500件 → 1000件

新データソースの追加（Box, Autodesk等）:
    → フィルタボタンを1つ追加するだけ（例：📐 Autodesk）
    → UIの構造は変わらない
```

低評価回答を確認 → 改善要望を燈チームに連携
```

**UX上のポイント**:
- **ROI可視化**：経営層が「投資対効果」を語れるデータ
- **エクスポート**：社内の報告フォーマットに合わせて加工可能
- **改善サイクル**：低評価 → 原因特定 → 改善のループを回せる

### 3-2. UX原則（弘電社案件特有の考慮事項）

| 原則 | 理由 | 実装方針 |
|------|------|---------|
| **Direct ファースト** | 現場では既にDirectを使っている。新しいアプリを入れさせない | チャットボットUIをメインに。管理画面はWeb別枠 |
| **片手で使える** | 現場作業中、スマホ片手で操作する場面が多い | ボタン・テンプレート中心。長文入力を強制しない |
| **根拠を示す** | 電気工事は安全に直結。AIの回答だけでは不安 | 全回答に根拠条文を付与。原文リンクを必ず提示 |
| **免責を明示** | AI回答の誤りが事故につながるリスク | 「重要な判断には原文をご確認ください」を常時表示 |
| **段階的開放** | 一度に全機能を見せると混乱する | まず規定検索のみ → 慣れたら社内文書 → 横断検索 |
| **フィードバックで育てる** | 回答精度はユーザーの声で改善される | 👍👎 を全回答に設置。低評価は自動で改善キューへ |

### 3-3. 利用定着のための施策

#### フェーズ別ロールアウト

```
Phase 1（導入月）
  ├─ 対象：工事部のパイロットチーム（20名程度）
  ├─ 機能：規定検索モードのみ
  ├─ 目標：1人あたり週3回以上の利用
  └─ 施策：利用促進メッセージ（週1回、AIボットから自動送信）

Phase 2（2〜3ヶ月目）
  ├─ 対象：工事部全体（180名）+ 営業部（85名）
  ├─ 機能：規定検索 + 社内文書モード追加
  ├─ 目標：月間質問数 500件以上
  └─ 施策：成功事例の社内共有、質問テンプレートの拡充

Phase 3（4ヶ月目〜）
  ├─ 対象：全社（634名）
  ├─ 機能：全モード開放 + 横断検索
  ├─ 目標：月間質問数 1,000件以上
  └─ 施策：部門別利用ランキング、経営会議での成果報告
```

#### 利用促進の仕掛け

| 施策 | 内容 | 期待効果 |
|------|------|---------|
| **週間おすすめ質問** | AIボットから毎週月曜に「今週のおすすめ質問」を配信 | 「こんなことも聞けるんだ」の気づき |
| **回答ブックマーク** | よく使う回答を保存できる機能 | リピート利用の促進 |
| **利用状況の可視化** | 「今月○件の質問に回答しました」を月末に通知 | 利用の習慣化 |
| **新着規定アラート** | 新しい書類がDBに追加された時に通知 | 最新情報へのアクセス動機 |

### 設計判断の一覧

| 設計判断 | 決定 | 理由 |
|---------|------|------|
| メインUI | Direct上のチャットボット | 現場で使い慣れたツール。新アプリ不要 |
| 検索の入口 | **1つだけ**（データソースボタン + 質問入力） | 入口を分けると迷う。質問文に書類名を含めれば絞り込める |
| データソース選択 | 5種のフィルタボタン | 300件単位ではなくDB単位で絞る。シンプル |
| 書類の絞込み | AIが自動判定 | 300件をユーザーに選ばせるのは非現実的 |
| 検索対象の管理 | Hikari管理画面でチェック（事前選択） | チェック数で精度・速度をコントロール |
| 回答フォーマット | 回答→根拠→アクションの3層カード | 信頼性と使いやすさの両立 |
| 検索した書類の表示 | 常に明示 | 透明性を確保し、見落としに気づける |
| 管理画面 | Hikari上のWebアプリ | 事前選択 + ROI可視化 + 精度管理 |
| フェーズ展開 | 建設DB+共通+個人 → +Box → +Autodesk | ボタンを段階的に追加するだけ |
