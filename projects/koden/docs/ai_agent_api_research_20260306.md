## 1. AIエージェント/RAGの基礎知識

### 1-1. AIエージェントとは

AIエージェントとは、従来のチャットボットのように会話ができるだけでなく、**タスクの計画・実行・判断までを自律的に行えるAIシステム**のこと。2025年後半から市場形成が本格化し、Gartnerは2026年までに40%の企業がエージェントスウォーム（協調型AIチーム）を活用すると予測している。

| 概念 | 説明 | 弘電社PJでの該当箇所 |
|------|------|---------------------|
| **LLM** | 大規模言語モデル。自然言語を理解・生成する基盤AI | 光/Hikari の建設特化LLM（AKARI Construction LLM） |
| **RAG** | 検索拡張生成。社内文書を検索し、その情報に基づいて回答を生成 | 建設DB・共通文書・内線規定等を参照した質問回答 |
| **AIエージェント** | LLM+RAG+ツール連携で自律的にタスクを遂行 | Direct連携ボット→光API→建設ナレッジ検索の一連フロー |
| **MCP** | Model Context Protocol。外部データソースとLLMを安全に接続する仕組み | Box連携、Autodesk連携で活用予定 |
| **OAuth 2.0** | トークンベースの認証方式 | フェーズ2でユーザー識別・他社展開のために実装予定 |

### 1-2. RAGの仕組みと精度課題

RAGは以下の3ステップで動作する：

1. **検索（Retrieval）**: ユーザーの質問に関連する文書をベクトル検索で取得
2. **拡張（Augmentation）**: 検索結果をプロンプトに組み込む
3. **生成（Generation）**: LLMが検索結果を踏まえて回答を生成

| 課題 | 内容 | 弘電社PJでの対応 |
|------|------|-----------------|
| 検索精度の低下 | 対象文書が増えると、無関係な文書がノイズとして混入 | フェーズ2 STEP1で50書類横断の精度検証を実施中 |
| ハルシネーション | もっともらしい誤回答の生成 | 検索箇所の根拠提示で信頼性を担保 |
| 文書のノイズ | OCR品質やフォーマットの問題 | フェーズ1でノイズ除去を実施済み |
| 横断検索の複雑性 | 複数DB間での検索ロジック構築が必要 | STEP1で建設DB×共通文書の横断検証中 |

> **商談での仮説**: 弘電社のフェーズ1検証では、内線規定・東京電力パワーグリッド・電気設備基準のいずれも精度100%を達成。ただし書類数が5→30→50と増加した際の精度維持が、フェーズ2 STEP1の最重要検証ポイントとなっている。

> **商談での活用**: 「精度100%の実績」はフェーズ2提案の強力な裏付け。横断検索の精度検証結果を早期に報告し、STEP2以降の本開発への信頼感を醸成する。
