## 1. HSコードの基礎知識

### 1.1 HSコードの構造
- **HS（Harmonized System）**: WCO（世界税関機構）が管理する国際共通の商品分類体系。212ヶ国・地域で使用
- **6桁が国際共通**（21部・96類・1,228項・5,612号）
- **日本は9桁**（6桁 + 3桁の統計品目番号）で運用
- 構造: `XX`（類/Chapter） → `XXXX`（項/Heading） → `XXXXXX`（号/Subheading） → `XXXXXXXXX`（統計品目番号）
- **5年ごとに大幅改正**あり（最新は2026年1月1日版）。旧コードでのNACCS申告はエラーになる

### 1.2 分類の基本ルール（通則/GRI）
品目分類は「通則（General Rules of Interpretation）」に従って決定される：

| 通則 | 内容 | ポイント |
|------|------|----------|
| **通則1** | 部・類の表題は参考。**項の規定**と部注・類注で決定 | 最も基本。まず項の記述を読む |
| **通則2(a)** | 未完成品・未組立品も完成品として分類 | - |
| **通則2(b)** | 混合物・複合材料は通則3で決定 | - |
| **通則3(a)** | 最も特殊な限定的記述の項に分類 | 具体的な記述が優先 |
| **通則3(b)** | **本質的特性（essential character）**を与える構成要素で分類 | 食品混合物で超重要 |
| **通則3(c)** | (a)(b)で決定できなければ、最も大きい番号の項 | 最終手段 |

### 1.3 本質的特性（Essential Character）の判定要素
通則3(b)で使われる「本質的特性」の判定基準（ケースバイケース）：
- **機能的支配性**: その構成要素がなければ製品が機能しないか
- **複雑性**: 技術的に最も重要な構成要素
- **相対的価値**: 構成要素間の価格比率
- **消費者認識**: 消費者がその製品の本質として何を認識するか
- **重量**: 一要素に過ぎず、自動的に決定されるものではない

> **商談での仮説**: 二葉の通関士は日々この判定を「経験と勘」で行っている可能性が高い。このノウハウを構造化できれば、AIの判定根拠として活用できる

### 1.4 食品分類における「20%ルール」（重要）
**第16類の注2**: 肉・魚・甲殻類等の含有量が**全重量の20%超**の場合 → 第16類に分類

| 含有量 | 分類先 |
|--------|--------|
| 肉・魚等 > 20% | **第16類**（肉・魚等の調製品） |
| 肉・魚等 ≤ 20% かつ穀物ベース | **第19類**（穀物の調製品） |
| 肉・魚等 ≤ 20% かつ上記以外 | **第20類**（野菜・果実等調製品）or **第21類**（各種調製食料品） |

**例外**: 第19.02項（パスタ・餃子等の詰物製品）、第21.03項（ソース）、第21.04項は別ルール

> **商談での仮説**: 海鮮チヂミの分類が難しかった原因はここにある可能性。海鮮の重量比率が20%前後だと、第16類と第19類の境界線上に位置し、AIが迷う

## 2. 食品・冷凍食品のHSコード分類（二葉の主戦場）

### 2.1 食品関連の類構造（第4部：調製食料品）

| 類 | 内容 | 代表的品目 |
|----|------|-----------|
| **第16類** | 肉・魚の調製品 | ソーセージ、缶詰、海鮮チヂミ（肉魚>20%の場合） |
| **第17類** | 糖類・砂糖菓子 | ショ糖、チョコレート（ココアなし） |
| **第18類** | ココア・その調製品 | チョコレート（ココアあり） |
| **第19類** | 穀物・ベーカリー製品 | パン、パスタ、餃子、ケーキ、ビスケット |
| **第20類** | 野菜・果実の調製品 | ジャム、果実ジュース、冷凍野菜 |
| **第21類** | 各種の調製食料品 | ソース、スープ、混合調味料、その他 |
| **第22類** | 飲料 | ジュース（調整あり）、アルコール |

### 2.2 二葉の検証商品の分類分析

#### 柚子100%果汁
- **正解**: 2009.31 or 2009.39（**柑橘類**果実ジュース）
- **AIの誤回答**: 2009.89-000（その他の果実ジュース）
- **分類のポイント**: 柚子が「柑橘類（citrus）」であることの認識が鍵
- **難しさ**: 柚子は日本特有の柑橘類で、海外のHS解説では明示されにくい。AIの一般知識では柚子を柑橘類と正確に紐付けられない可能性

> **仮説**: 税関の事前教示事例や勉強会資料に「柚子＝柑橘類」の分類根拠が記載されているはず。これをRAGに組み込めば解決可能

#### 海鮮チヂミ
- **候補が複数**: 第16類（肉魚>20%）vs 第19類（穀物ベース）
- **分類のポイント**: 海鮮の重量比率が20%を超えるか否かで分類先が変わる
- **難しさ**: 製品の成分比率情報がないとAIは判定できない。また「チヂミ」という料理名からだけでは成分比率は推測困難

> **仮説**: 実務では商品スペックシート（成分表・重量比率）を参照して判定しているはず。AIにもこの情報を入力させるステップが必要

#### レーズン入りバターロール
- **分類**: 1905.90（パン・ベーカリー製品）
- **凍結の場合**: 1905.90.10.49（冷凍パン・ベーカリー製品）
- **比較的容易**: パン類は第19類のベーカリー製品として明確

#### ビーフカレー
- **分類のポイント**: 牛肉の含有量が20%超なら第16類、以下なら第21類
- **難しさ**: カレーの場合、肉の比率は製品によって大きく異なる

#### チョコ＆ビスケット
- **分類**: 1905.31（甘いビスケット）が基本
- **ココアコーティングの有無**で細分類が変わる
- **難しさ**: 「チョコレートを含むビスケット」と「ビスケットを含むチョコレート」で分類先が異なる可能性

#### アサイードリンク
- **100%ジュース**: 2009.90（混合果実ジュース）
- **加糖・加水あり**: 2202.99（果汁ベース飲料）
- **ビタミン強化品**: 2106.90（食品調製品）
- **難しさ**: 製品の組成（果汁含有率、添加物の有無）で3つ以上の候補に分かれる

### 2.3 冷凍食品分類の特殊な難しさ

1. **加工度の判断**: 「調理済み」vs「未調理」で分類先が変わる
2. **温度帯による区別**: 冷凍であること自体が細分類に影響する場合がある
3. **成分比率の重要性**: 混合食品では肉・魚の重量比率20%が境界線
4. **製品名からの推測困難**: 「海鮮チヂミ」「ビーフカレー」は成分比率が製品ごとに異なる
5. **原材料の正確な把握**: 食品添加物・保存料なども分類に影響する場合がある

## 3. 通関業務の実務とHSコード判定フロー

### 3.1 通関士のHSコード判定プロセス（想定）
1. インボイス・パッキングリストから品目情報を取得
2. 商品スペック（成分・製法・用途・形状）を確認
3. 関税率表（実行関税率表）を参照して候補を絞り込み
4. 類注・項の規定・通則を適用して分類を確定
5. 過去の事前教示事例・分類例規で裏付け確認
6. NACCS（通関情報処理システム）に統計品目番号を入力して申告

### 3.2 判定で参照する資料群
- **実行関税率表**: 税関HPで公開（2026年1月1日版が最新）
- **関税率表解説**: 各項の詳細な解説書
- **分類例規**: 税関が公表する分類事例
- **事前教示回答**: 税関に照会した結果のデータベース（Web検索可能）
- **WCO解説書（Explanatory Notes）**: 国際的な解釈指針
- **社内ナレッジ**: 勉強会資料、過去の判定実績

> **仮説**: 二葉の通関士（島本さん）はこれらの資料を横断的に参照して判定している。この「参照プロセス」をRAGで再現することがAIエージェントの核心

### 3.3 事前教示制度
- 税関に事前に品目分類を照会できる制度
- 文書・メール・窓口・電話で照会可能
- 回答はデータベースで公開（他社の事例も参照可能）
- 税関HP: https://www.customs.go.jp/tetsuzuki/bunrui/index.htm

> **仮説**: 事前教示のデータベースは膨大な「正解データ付き判定事例」の宝庫。これをRAGのデータソースとして活用すれば精度が大幅に向上する可能性

### 3.4 食品輸入特有の手続き
食品輸入は通常の通関に加えて以下が必要：
1. **食品衛生法に基づく輸入届出**（検疫所への届出義務）
2. **植物防疫法**（農産物の場合）
3. **動物検疫法**（畜産物の場合）
4. HSコード判定 → 検疫所審査 → 税関申告の一連のフロー

> **仮説**: HSコードの判定ミスは、関税額の誤りだけでなく、食品衛生法上の手続きにも影響する。冷凍食品では特にこのリスクが高い

## 4. HSコード判定のAI活用（競合・先行事例）

### 4.1 国内の主要事例

#### NEC HSコード特定支援システム（2024年9月発表）
- 生成AIを活用してHSコード候補を提示
- アイテム名入力 → HSコード候補提示 → 不足情報をAIが提示して特定を支援
- 通関士不足への対応が目的
- 2025年度からの提供開始予定
- 特許出願中

> **仮説**: NECが参入していることは、市場のvalidationになる。一方で、NECのソリューションは汎用的であり、**冷凍食品に特化した精度**では差別化可能

### 4.2 海外の主要サービス

| サービス | 特徴 |
|----------|------|
| **Zonos** | ECに特化したHS分類。AIベース |
| **3CE Technologies** | HSコード分類のAIプラットフォーム |
| **Avalara** | 税務・関税の総合ソリューション |
| **Tarifflo** | AI関税分類ツール |
| **WCO HSトレーナー** | WCO公式の研修ツール |

### 4.3 AI分類の技術的アプローチ

| アプローチ | 特徴 | 精度課題 |
|-----------|------|----------|
| **テキスト分類（NLP）** | 商品説明からHS6桁を推定 | 類似商品の区別が困難 |
| **RAG** | 関税率表・事前教示事例を参照して回答 | データの質と構造化が鍵 |
| **ルール+AI** | 通則・類注をルール化+AIで候補絞込 | ルールの網羅性がボトルネック |
| **マルチステップ推論** | 類→項→号の段階的絞込 | 途中の判断ミスが最終結果に影響 |

> **仮説**: 二葉向けのアプローチは「**マルチステップRAG + ルールベース検証**」が最適。
> 1. 商品情報から類を推定（NLP）
> 2. 類注・20%ルール等のルール適用（ルールベース）
> 3. 事前教示事例・勉強会資料で裏付け（RAG）
> 4. 根拠を明示して通関士に提示（Human-in-the-loop）

### 4.4 AIの精度課題とハルシネーション対策

**AIが間違えやすいパターン**:
- 類似商品の微妙な区別（柚子を柑橘類と認識できない等）
- 成分比率による分類境界線上の商品
- 日本特有の食品（チヂミ、もち、和菓子等）
- 加工度の判断（半調理品など）

**ハルシネーション対策**:
1. **根拠の明示**: 「関税率表第X類注Y項に基づき…」と出典を必ず提示
2. **確信度スコア**: 判定の確実性をスコアで表示し、低スコアは通関士に回付
3. **RAGによるGrounding**: 関税率表・事前教示事例のみを参照し、モデルの一般知識に依存しない
4. **段階的判定**: 類→項→号を段階的に絞り込み、各ステップで根拠を確認

## 5. 通関業界の動向と二葉の位置づけ

### 5.1 株式会社二葉の概要
- **事業**: 冷凍物流・通関業（港湾事業者）
- **規模**: 資本金6.3億円、売上170億円（2024年3月期）、従業員302名（グループ660名）
- **キャッチコピー**: 「みなとから日本の食をささえる。」
- **グループ**: 二葉物流システム（関西事業も2025年開始）
- **拠点**: 大井、東扇島、南港、横浜支店 他
- SDGsパートナー登録（国交省、2022年）

### 5.2 通関業界の構造的課題

1. **通関士不足**: 生産年齢人口の減少、専門知識の習得に長期間必要
2. **業務の複雑化**: EPA/FTA拡大、経済安全保障対応、HSコード改正
3. **中小業者の経営圧迫**: 人件費上昇 vs 通関料金への転嫁困難
4. **レガシーシステム**: NACCSは稼働中だが紙ベース作業が残存
5. **2024年問題**: ドライバー不足がコールドチェーンに直接影響

### 5.3 コールドチェーンと通関のスピード要求
- 冷凍食品は通関が遅れると品質リスク（温度逸脱）
- 冷蔵倉庫の保管コストは常温の2-3倍
- 迅速な通関 = コスト削減 + 品質維持

> **仮説**: 二葉にとってHSコード判定の効率化は「精度向上」だけでなく「スピードアップ」の観点でも重要。冷凍食品は通関のリードタイム短縮が直接的にコスト削減と品質維持につながる

### 5.4 政府の動き
- **スマート税関構想2027**: 税関のDX推進
- **貿易DXアクションプラン（2024年6月）**: 経産省が策定
- **NACCS近代化**: 第6次NACCS（2017年〜）稼働中
- **官民合同検討会（2025年6月〜）**: 貿易手続デジタル化の議論
