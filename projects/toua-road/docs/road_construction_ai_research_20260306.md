## 1. 道路建設業界のAI/DXの基礎知識

### 1-1. 国の政策フレームワーク：i-Construction 2.0

国土交通省は2024年4月に「i-Construction 2.0」を発表し、建設現場のオートメーション化を推進している。

| 項目 | 内容 |
|------|------|
| 目標年次 | 2040年度 |
| 省人化目標 | 少なくとも3割 |
| 生産性目標 | 1.5倍向上 |
| 3つの柱 | 施工のオートメーション化 / データ連携のオートメーション化 / 施工管理のオートメーション化 |
| ICT施工原則化 | 2025年度から原則化、施工データ取得環境を徹底 |
| 舗装工のICT原則化 | 2026年度から発注者指定型拡大、2027年度以降原則化 |
| 積算システム改良 | 2026年度目標で3次元モデルから数量を直接積算に活用 |
| BIM/CIM義務化 | 2023年度から直轄工事で原則適用、2029年に全国展開 |

> **商談での活用**: 東亜道路工業は道路舗装大手として、2026年度からのICT舗装工原則化の波に直面する。この文脈で「AIによる実行予算書活用」は、ICT施工データと組み合わせることでさらなる価値が出る可能性がある。

### 1-2. 建設業界のAI活用実態

アンドパッドの2026年2月調査（建設業従事者2,000人対象）によると：

| 指標 | 数値 |
|------|------|
| AI活用率（積極的+試験的） | 34.8% |
| 活用予定なし | 47.3% |
| AI導入目的1位 | 省力化/作業効率化（39.7%） |
| AI導入目的2位 | 人手不足への対応（33.8%） |
| AI導入目的3位 | 品質の安定化/ミス削減（30.3%） |
| AI導入目的4位 | 安全性の向上/リスク低減（20.3%） |
| 建設・不動産業のAI活用割合 | 9.4%（全業種最低水準） |
| 導入企業の効果実感率 | 約9割 |

> **商談での仮説**: 東亜道路工業はDX推進部を設置し、燈のAIエージェントPoCを実施している時点で、業界内では先進的な取り組みをしている。「全業種最低の9.4%」という数字は、逆に先行者優位を取れる根拠として使える。

### 1-3. 建設業の構造的課題

| 課題 | 詳細 |
|------|------|
| 2024年問題 | 時間外労働の上限規制が建設業にも適用（2024年4月〜） |
| 技術継承 | ベテラン社員への依存、5〜10年以内の大量退職リスク |
| 人手不足 | 生産年齢人口の減少、採用難 |
| 社内データの埋没 | Excel書類等の財産が若手の実務に生かされていない |
| 重層下請構造 | 労働生産性の低さの構造的要因 |

> **商談での活用**: 2/26の会議で東亜道路工業側も「2024年問題」「技術継承」「過去データのブラックボックス化」を自社課題として認識済み。このPJはまさにこれら3課題の解決に直結する。

## 2. 東亜道路工業の業務におけるAI活用ポイント

### 2-1. PJ概要（商談から得た情報）

| 項目 | 内容 |
|------|------|
| PJ名称 | 工事費用概算算出エージェント |
| 期間 | 2026年1月〜3月（検証フェーズ） |
| 目的 | 過去の実行予算書データを活用し、見積書・実行予算書の作成を支援 |
| 対象データ | 実行予算書83件（構造化完了） |
| 使用モデル | GPTシリーズ |
| 先方メンバー | DX推進部：山野氏、霧氏、塚本部長 |
| 想定ユーザー | 営業部100名＋技術部50名、将来的に全現場250名規模 |

### 2-2. 検証済み機能と精度

| 機能 | 検証結果 | 精度 |
|------|----------|------|
| 1. 類似工事の検索 | 全4問中4問正解 | 100% |
| 2. 工期の回答 | 実施工期を正確に取得 | 合格 |
| 3. 単価の検索 | 全2問中2問正解 | 100% |
| 4. 直接/間接工事費の内訳取得 | 項目の網羅的列挙に成功 | 合格 |
| 5. 中央値等の計算 | LLMのハルシネーション発生 | 要改善（外部ツール対応） |
| 6. 機能の使い分け | 次回（3/13）検証予定 | 未検証 |
| 部掛かりデータの取得 | 表形式での出力に成功 | 合格（日数列追加対応中） |

### 2-3. 東亜道路工業の業務フローとAI適用箇所

```
[受注] → [見積書作成] → [実行予算書作成] → [施工] → [実績フィードバック]
              ↑                    ↑
         AIエージェント        AIエージェント
         (類似工事検索)        (部掛かり・単価参照)
```

| 業務 | 現状の課題 | AIによる解決 |
|------|-----------|-------------|
| 見積書作成 | 経験値に依存、人によってバラつき | 過去類似工事の自動検索・単価参照 |
| 実行予算書作成 | 若手は項目の抜け漏れが発生 | 講種別の部掛かり表を自動提示 |
| 工事費概算 | ベテランの暗黙知に依存 | 中央値・範囲値での参考情報提示 |
| 間接費の見積もり | 必要項目の見落としリスク | 類似工事の間接費一覧を自動取得 |

> **商談での仮説**: 「単価を変えるだけで済むような実行予算書の下書きを作成できること」が最終的なゴール（1/30会議での発言）。現在の検証結果はこのゴールに向けて着実に前進している。

## 3. 道路建設の実務とAI化

### 3-1. 実行予算書の構造

商談から得た実行予算書の構造：

| 要素 | 色分け | 説明 |
|------|--------|------|
| 工事情報（鏡） | - | 工事名、支店、工種、受注金額、現場代理人、監理技術者 |
| 講（大分類） | オレンジ | 例：表層工、基層工、路盤工 |
| 講種（中分類） | ピンク | 例：舗版撤去工（数量・平米を入力） |
| 項目（小分類） | 薄い黄色 | 材料・人工・機械の明細（数量・単価・金額） |
| 水色 | 水色 | 機械等の「ずっといる」項目 |

### 3-2. 構造化データベース設計（本PJで構築済み）

| データベース | 格納内容 | 件数 |
|-------------|---------|------|
| 工事案件情報DB | 工事名、支店、工種、受注金額、契約、工事番号、工事種別、現場代理人、監理技術者 | 83件 |
| 直接工事費明細DB | 講別・裁別の項目名、規格、数量、単位、単価、金額 | 83件分の全行 |
| 間接費DB | 共通仮設費・現場経費の項目、数量、単価 | 83件分の全行 |

### 3-3. 金額を左右する変数（商談で確認済み）

| 変数 | 影響 |
|------|------|
| 工事数量（平米等） | 直接的にコストに影響 |
| 場所（支店・地域） | 単価の地域差 |
| 時期（年度） | 物価変動（直近3年で約2割上昇） |
| 規模（受注金額） | 間接費率等が変動 |
| 工事種別（アルファベット） | 都道府県/市町村/民間等の区分 |

### 3-4. 部掛かり（歩掛かり）の重要性

| 項目 | 内容 |
|------|------|
| 定義 | 国交省の積算基準に基づく、単位施工量あたりの労務・機械・材料の標準使用量 |
| 用途 | 「1日あたり何平米施工できるか」を算出し、工期・コストを見積もる |
| データソース | 実行予算書の備考欄（日数×人数の記載） |
| 課題 | 書き方が人によりバラバラ、記載がないケースもある |
| AIの役割 | 過去の類似工事から部掛かりを検索し、日当たり施工量を提示 |

> **商談での活用**: 部掛かりは若手が最も苦手とする領域。ベテランは「頭の中に入っている」ため使わないが、若手がAIで部掛かりを引き出し、課長がレビューするワークフローが理想。

### 4-2. 本PJの競合優位性

| 比較軸 | 既存サービス | 本PJ（燈×東亜道路工業） |
|--------|------------|----------------------|
| データソース | 汎用的な積算基準 | 自社の実行予算書83件（実績データ） |
| 対象業種 | 建築・内装が中心 | 道路舗装に完全特化 |
| 出力形式 | 数量・コスト | 類似工事検索＋部掛かり＋単価＋工期の統合回答 |
| カスタマイズ性 | 汎用SaaS | 東亜道路工業の業務フローに最適化 |
| 技術継承 | 対応なし | ベテランの暗黙知をデータ化し若手に提供 |

> **商談での仮説**: 道路舗装に特化した「実行予算書AIエージェント」は市場に存在しない。自社データを活用した独自ソリューションという点で、既存の積算AIとは根本的に異なるポジショニングが取れる。
