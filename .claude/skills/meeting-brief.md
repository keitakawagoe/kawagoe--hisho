# 商談ブリーフィングシート生成スキル

商談のアポが確定した際に、必要な情報を自動収集し、構造化されたブリーフィングシートを生成する。

---

## 1. 概要

商談前に必要な情報を一枚にまとめたブリーフィングシートを生成するスキル。
以下の情報ソースから自動的に情報を収集・統合する：

- **Googleカレンダー** → 日時・参加者・Meet URL
- **Gmail** → 過去のやり取り・商談回数の推定
- **プロジェクトフォルダ** → README.md・minutes/・proposals/ から案件コンテキスト
- **Web検索** → 企業売上・従業員数・事業内容

## 2. 使用場面

- 商談のアポが確定したとき
- MTG前日の準備確認
- 社内への商談共有・引き継ぎ

## 3. 実行手順

### STEP 1: カレンダーからMTG情報を取得

```bash
.venv/bin/python scripts/gcal.py range {開始日} {終了日}
```

対象MTGを特定し、以下を抽出：
- 日時、Meet URL
- 参加者（メールアドレス → 氏名・所属の特定に使用）

### STEP 2: プロジェクトフォルダの確認

```
projects/{案件スラッグ}/README.md    → 案件概要・ステークホルダー・経緯
projects/{案件スラッグ}/emails/      → 過去メールから商談回数を推定
projects/{案件スラッグ}/minutes/     → 過去の商談メモ
projects/{案件スラッグ}/proposals/   → 提案状況
```

プロジェクトフォルダが存在しない場合は、STEP 3のGmail検索結果をもとに新規作成を提案する。

### STEP 3: Gmailから過去のやり取りを検索

```bash
.venv/bin/python scripts/gmail.py search "{企業名 or 担当者名}"
```

- メールの件数・時系列から商談回数を推定
- 直近のメールから現在の論点・温度感を把握

### STEP 4: 企業情報をWeb検索

WebSearchツールで以下を検索：
- `{企業名} 売上高 従業員数 {直近年度}`
- 企業HPの会社概要ページをWebFetchで取得

### STEP 5: ブリーフィングシート生成

以下のテンプレートに情報を埋めて出力する。

## 4. 出力テンプレート

出力先: `projects/{案件スラッグ}/docs/meeting_brief_{YYYYMMDD}.md`

**フォーマットルール:**
- Slack投稿を想定し、テーブル（表）は使わない
- 太字見出し + 箇条書きで構成する
- 先方の氏名には必ず「様」を付ける（例：井上様、矢野様）
- 燈側メンバーには「様」は不要

```markdown
**{企業名}【{M/D}】**

**企業売上**
{売上高}（{年度}）

**カウンターの部署/役職/氏名**
{部署} / {役職} / {氏名}様
※ 役職が不明な場合は「（役職未確認）」と明記

**何回目商談か**
{N}回目（過去: {日付と概要を箇条書き}）

**商談のgoal**
{一文でゴールを表現}
- {具体的なゴール1}
- {具体的なゴール2}
- {具体的なゴール3}

**目標**
- {MTG終了時に確定させたいアクション1}
- {MTG終了時に確定させたいアクション2}
- {MTG終了時に確定させたいアクション3}

**想定契約金額**
{金額}（{スコープの概要}）

**想定アジェンダ**
1. {時間} {内容}（{目的}）
2. {時間} {内容}（{目的}）
3. ...

**背景情報**
- {既存取引の状況}
- {提案の経緯}
- {先方の温度感}
- {リスク・注意点}
```

## 5. ゴール設定の考え方

商談のゴールは「MTG終了時にどういう状態になっていたいか」で設定する。
以下の要素を必ず含める：

- **合意事項**: 何について合意を取るか
- **確認事項**: 何を確認・ヒアリングするか（特に稟議フロー・決裁者）
- **次のアクション**: MTG後に双方が何をするか（期限付き）

### 商談フェーズ別のゴール例

| フェーズ | 典型的なゴール |
|---------|--------------|
| 初回ヒアリング | 課題の具体化 + 方向性合意 + 稟議フロー把握 + 次アクション確定 |
| 提案説明 | 提案内容への合意 + 見積承認 + 稟議スケジュール確認 |
| 決裁者プレゼン | 予算承認 + 契約手続きの開始合意 |
| PoC結果報告 | 成果の確認 + 本開発への移行合意 |

## 6. 想定契約金額の扱い

- 金額が確定している場合 → ブリーフィングシートに記載
- 未確定の場合 → 「未定（概算: {金額帯}）」と記載
- **スライド資料には金額を入れるかどうかを都度ユーザーに確認する**

## 7. 注意事項

- 先方の氏名には必ず「様」を付ける
- カウンターの役職が不明な場合は必ず「（役職未確認）」と明記する。推測で埋めない
- 商談回数はGmail・カレンダー・minutes/の情報を突き合わせて正確にカウントする
- 企業売上はWeb検索で最新の公開情報を取得する。非上場で情報がない場合は「非公開」と記載
- ブリーフィングシート生成後、ユーザーに内容を確認してもらう
- 情報が不足している項目は空欄にせず「（要確認）」と明記する
- テーブル（表）は使わず、太字見出し + 箇条書きで表現する（Slack互換）
