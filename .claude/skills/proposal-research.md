# 提案準備リサーチスキル

商談・提案に向けて、業界動向・競合分析・技術調査を行い、仮説付きの調査レポートを生成する。

---

## 1. 概要

案件の提案準備として、対象テーマに関する包括的なリサーチを行うスキル。
単なる情報収集ではなく、**商談で使える仮説とヒアリング項目**まで落とし込む。

今回のHSコード調査レポート（二葉案件）と同等の粒度・網羅性を持つレポートを生成する。

## 2. 使用場面

- 新規案件の初回商談前（業界・顧客・技術の理解が必要な場合）
- 提案の方向性を固める前のファクト収集
- 技術検証（PoC）のスコープを決めるための先行事例調査
- 既存案件で新たなテーマ・深掘り領域が発生した場合

## 3. インプット

### 必須
- **案件スラッグ**: `projects/{案件スラッグ}/` のフォルダ名
- **調査テーマ**: 自然言語で指定（例: 「HSコード判定のAI活用」「在庫需要予測AI」「図面OCR」）

### 自動収集（案件フォルダから読み取り）
以下を **すべて読み込んでから** リサーチを開始する:

| ソース | パス | 読み取る情報 |
|--------|------|-------------|
| **案件概要** | `README.md` | 顧客情報・ステークホルダー・案件経緯・課題感 |
| **商談メモ** | `minutes/*.md` | 顧客の発言・温度感・具体的な課題・要望 |
| **メール** | `emails/*.md` | やり取りの経緯・先方の関心事項・制約条件 |
| **既存資料** | `docs/*.md` | 過去の調査結果・PoC結果など（あれば） |
| **検証進捗** | `verification/PROGRESS.md` | 技術検証の状況（あれば） |

**重要**: README だけでなく、minutes/ と emails/ を丁寧に読むことで、顧客が実際に口にした課題・言い回し・懸念を把握する。これがレポートの仮説の質を左右する。

## 4. 実行手順

### STEP 1: 案件コンテキストの把握

```
projects/{案件スラッグ}/README.md
projects/{案件スラッグ}/minutes/     （全ファイル）
projects/{案件スラッグ}/emails/      （全ファイル）
projects/{案件スラッグ}/docs/        （既存調査があれば）
projects/{案件スラッグ}/verification/PROGRESS.md （あれば）
```

読み取り後、以下を整理する:
- 顧客の業種・規模・主力事業
- 顧客が抱える課題（顧客自身の言葉で）
- 商談の経緯・現在のフェーズ
- 次回MTGの目的・日時
- 技術検証の状況（あれば）

### STEP 2: 基礎知識の調査

調査テーマに関する **業界の前提知識** をWeb検索で収集する。

調査観点:
- テーマの基本的な仕組み・体系・用語
- 業界の構造・主要プレイヤー
- 規制・法制度・標準規格
- 市場規模・トレンド（数字付き）

検索クエリ例:
- `{テーマ} 基礎知識 仕組み`
- `{テーマ} 業界 市場規模 {現在の年}`
- `{テーマ} 規制 法律 日本`

### STEP 3: 顧客の業務分析

STEP 1で把握した顧客の課題を深掘りし、業務プロセスの中でテーマがどう位置づけられるか調査する。

調査観点:
- 対象業務の一般的なフロー（ステップバイステップ）
- ボトルネック・課題が発生するポイント
- 業務で参照する資料・システム
- 関連する他の業務プロセス（上流・下流）
- 業界特有の数字（件数・時間・コスト）

### STEP 4: 競合・先行事例の調査

テーマに関連するAI/DXソリューションの先行事例を調査する。

調査観点:
- **国内の主要事例**（プレスリリース・ニュース記事）
- **海外の主要サービス**（SaaS・ツール）
- 各社のアプローチ（技術手法）・精度・価格帯
- 顧客の競合企業のDX動向（あれば）

検索クエリ例:
- `{テーマ} AI {現在の年} 事例`
- `{テーマ} DX ソリューション`
- `{テーマ} AI accuracy benchmark`
- `{顧客の業界} AI 導入事例`

### STEP 5: 技術動向の調査

AI/技術面でのアプローチ・精度・課題を調査する。

調査観点:
- 技術的なアプローチの分類（RAG、ファインチューニング、ルールベース等）
- 各アプローチの精度・課題・適用条件
- 学術論文・ベンチマーク（arXiv等）
- ハルシネーション対策・精度向上手法
- 業界のリスク動向（罰金・規制強化等）

検索クエリ例:
- `{テーマ} LLM benchmark arXiv {現在の年}`
- `{テーマ} AI accuracy challenges`
- `{テーマ} RAG approach`

### STEP 6: 仮説の構築とレポート生成

STEP 1〜5の情報を統合し、以下のレポートを生成する。

## 5. アウトプット

### 出力先


## 6. 仮説の書き方ルール

仮説セクション（セクション6）は本スキルの最重要アウトプット。以下のルールに従う:

### 仮説の構成要素
- **仮説文**: 「〜である」「〜の可能性が高い」の断定形で書く
- **確認ポイント**: 商談で何を聞けばこの仮説が検証できるかを明記
- **根拠**: レポートのどのセクションの情報に基づく仮説かを `>` 引用ブロックで補足

### 仮説のカテゴリ
| カテゴリ | 内容 | 目安数 |
|---------|------|--------|
| **A. 業務課題** | 顧客の現場で起きている課題の仮説 | 5〜8個 |
| **B. 技術ソリューション** | 燈が提案する技術的アプローチの仮説 | 4〜6個 |
| **C. ビジネス** | 市場性・ROI・競合優位性の仮説 | 3〜5個 |
| **D. 深掘り質問** | 仮説を検証するための具体的な質問 | 8〜12個 |

### 仮説の品質基準
- **顧客の言葉に紐付いていること**: minutes/ や emails/ の顧客発言に基づく仮説を優先
- **数字で裏付けられていること**: 業界データ・ベンチマーク等の定量情報を根拠にする
- **反証可能であること**: 商談で「YES/NO」で確認できる粒度にする
- **actionableであること**: 仮説が正しかった場合、提案にどう反映するかが明確

## 7. レポート全体の品質基準

- **> 引用ブロック** を各セクションの最後に配置し、「商談での仮説」「商談での活用」として提案への示唆を記載する
- **テーブル（表）** を多用して情報を構造化する（Slack用ではなくMarkdownファイルなので表OK）
- **数字を必ず含める**: 市場規模、件数、精度、コスト等。数字のない仮説は弱い
- **出典URL** を参考リンク集にすべて記載する
- **顧客固有の分析** を必ず含める: 汎用的な業界レポートではなく、顧客の具体的な商品・サービス・課題に当てはめた分析をする
- **競合との差別化** を明記する: 燈のアプローチが先行事例・競合とどう異なるかを仮説として提示する

## 8. 注意事項

- 案件フォルダ（README, minutes, emails）を **すべて読んでから** Web検索を開始する。顧客コンテキストなしの汎用調査は禁止
- 調査テーマが広すぎる場合、ユーザーに確認してスコープを絞る
- 金額・技術選定に関する仮説は、確度が低い場合は「要確認」と明記する
- レポート生成後、ユーザーに内容を確認してもらう
- 既存の調査レポート（docs/ 内）がある場合は、重複を避けて差分情報を追記する方向で検討する
- Web検索は日本語・英語の両方で行い、国内外の情報を網羅する
