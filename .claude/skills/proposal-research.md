# 提案準備リサーチスキル

**目的: 顧客の業務のドメイン知識を深く理解し、商談当日に仮説をたくさん持って話せるようにする。**

---

## 1. 概要

このスキルの本質は **「顧客の業務を自分ごととして理解するための調査」** である。

**NG**: 業界トレンドや競合サービスの一覧を並べただけの表面的なレポート
**OK**: 「この業務はこういう仕組みで動いていて、ここが難しくて、だからこういう課題があるはず」と自分の言葉で語れるレベルのドメイン知識

### 調査の優先順位（重要）
1. **顧客の業務そのものの理解**（最重要・レポートの6割以上をここに割く）
   - その業務は具体的にどう動いているのか？
   - どんな書類・データ・システムを使っているのか？
   - なぜ難しいのか？どこに判断が必要なのか？
   - ベテランと若手で何が違うのか？
2. **業務課題の構造化** — ドメイン知識に基づいて「ここが課題のはず」という仮説を立てる
3. **競合・先行事例** — あくまで補足。業務理解が浅い状態で競合を並べても意味がない

### 良いレポートの基準
- レポートを読んだ後、顧客の担当者と **業務の具体的な話** ができる
- 「〜って、こういう仕組みですよね？」「〜の部分が大変だと思うんですが」と自分から切り出せる
- 顧客が「よくわかってるね」と思うレベルの業務理解

## 2. 使用場面

- 新規案件の初回商談前（顧客の業務を理解したい）
- 提案の方向性を固める前（課題の解像度を上げたい）
- 技術検証（PoC）のスコープを決める前（業務の中で何が一番難しいかを知りたい）
- 既存案件で新たなテーマ・深掘り領域が発生した場合

## 3. インプット

### 必須
- **案件スラッグ**: `projects/{案件スラッグ}/` のフォルダ名
- **調査テーマ**: 自然言語で指定（例: 「HSコード判定のAI活用」「在庫需要予測AI」「図面OCR」）

### 自動収集（案件フォルダから読み取り）
以下を **すべて読み込んでから** リサーチを開始する:

| ソース | パス | 読み取る情報 |
|--------|------|-------------|
| **案件概要** | `README.md` | 顧客情報・ステークホルダー・案件経緯・課題感 |
| **商談メモ** | `minutes/*.md` | 顧客の発言・温度感・具体的な課題・要望 |
| **メール** | `emails/*.md` | やり取りの経緯・先方の関心事項・制約条件 |
| **既存資料** | `docs/*.md` | 過去の調査結果・PoC結果など（あれば） |
| **検証進捗** | `verification/PROGRESS.md` | 技術検証の状況（あれば） |

**重要**: README だけでなく、minutes/ と emails/ を丁寧に読むことで、顧客が実際に口にした課題・言い回し・懸念を把握する。これがレポートの仮説の質を左右する。

## 4. 実行手順

### STEP 1: 案件コンテキストの把握

```
projects/{案件スラッグ}/README.md
projects/{案件スラッグ}/minutes/     （全ファイル）
projects/{案件スラッグ}/emails/      （全ファイル）
projects/{案件スラッグ}/docs/        （既存調査があれば）
projects/{案件スラッグ}/verification/PROGRESS.md （あれば）
```

読み取り後、以下を整理する:
- 顧客の業種・規模・主力事業
- 顧客が抱える課題（顧客自身の言葉で）
- 商談の経緯・現在のフェーズ
- 次回MTGの目的・日時
- 技術検証の状況（あれば）

### STEP 2: ドメイン知識の深掘り調査（最重要・最も時間をかける）

**このSTEPがレポートの質を決める。** 顧客の業務を「自分が担当者だったら」のレベルで理解するための調査。

#### 2a. 業務の仕組み・体系を理解する
- その業務は具体的にどういう **手順** で進むのか？（ステップバイステップ）
- どんな **書類・データ・システム** を使っているのか？
- 業務で使う **専門用語** は何か？（用語を正しく使えないと顧客に信頼されない）
- **分類体系・ルール** がある場合、その構造は？（例: HSコードの通則、積算の歩掛かり）
- **判断基準** は何か？（どこに人間の経験・知識が必要か）

#### 2b. なぜ難しいのか？を深掘りする
- ベテランと若手で **何が違うのか** ？（知識？経験？判断力？）
- **間違えやすいポイント** はどこか？
- **例外・境界ケース** はどんなものがあるか？
- 間違えた場合の **リスク・影響** は？（罰金？遅延？品質問題？）

#### 2c. 周辺業務との関係を理解する
- この業務の **上流**（何がインプットになるか）と **下流**（アウトプットが何に使われるか）
- 関連する **法制度・規制・業界基準**
- 業界特有の **商慣習・暗黙のルール**

検索クエリ例（業務理解を優先）:
- `{業務名} やり方 手順 初心者`
- `{業務名} 難しい ポイント コツ`
- `{専門用語} とは わかりやすく`
- `{業務名} ベテラン 若手 違い`
- `{業務名} ミス 事例 失敗`
- `{業界} {業務名} 基準書 マニュアル`

### STEP 3: 顧客固有の業務分析

STEP 2の一般知識を、STEP 1で把握した **顧客の具体的な状況** に当てはめる。

- 顧客の商品・サービスに当てはめると、業務のどこが特に難しいか？
- 商談・メールで言及された具体的な課題は、業務フローのどのステップで発生しているか？
- 顧客の規模・業種特性から、どんな固有の課題があるか？

### STEP 4: 競合・先行事例の調査（補足的位置づけ）

業務理解が十分にできた上で、AI/DXソリューションの先行事例を調査する。
**業務理解なしに競合を並べるのは禁止。** 競合の評価は「顧客の業務課題を解決できるか」の観点で行う。

調査観点:
- **国内の主要事例**（プレスリリース・ニュース記事）
- **海外の主要サービス**（SaaS・ツール）
- 各社のアプローチ（技術手法）・精度・価格帯
- **競合が解決できていない課題は何か？**（燈の差別化ポイント）

### STEP 5: 仮説の構築とレポート生成

STEP 1〜4の情報を統合し、以下のレポートを生成する。
**仮説は必ずドメイン知識（STEP 2）に裏付けられていること。** 業務を理解していない仮説は価値がない。

## 5. アウトプット

### 出力先

```
projects/{案件スラッグ}/docs/{テーマスラッグ}_research_{YYYYMMDD}.md
```

テーマスラッグは調査テーマを英語のスネークケースに変換（例: `hs_code_research`、`demand_forecast_research`）

### レポート構成

以下の7セクション構成で出力する。**セクション1〜3がレポートの核（全体の6割以上）。** セクション4〜5は補足。

```markdown
# {調査テーマ} 調査レポート
**作成日**: YYYY/MM/DD
**目的**: {次回MTGの日付} {顧客名} {担当者名}とのMTGで仮説をもって議論するための事前調査

---

## 1. {テーマ}の基礎知識（ドメイン知識）
（3〜5ページ相当。ここを厚く書く）

### 1.1 {テーマの全体像・構造}
- この業務/テーマは何のためにあるのか？
- 全体の体系・分類構造
- 主要な専門用語の解説（商談で使えるレベル）

### 1.2 {業務の具体的な仕組み}
- ステップバイステップの業務手順
- 各ステップで使う書類・データ・システム
- 判断が必要なポイントとその判断基準

### 1.3 {テーマの重要ルール・原則}
- 業務上重要なルール・基準
- 例外・境界ケース
- 間違えた場合のリスク・影響

### 1.4 なぜ難しいのか

### 3.3 業界の標準的なやり方と顧客のやり方の差

> **商談での仮説**: {実務理解に基づく具体的な仮説}
```

## 6. 仮説の書き方ルール

仮説セクション（セクション6）は本スキルの最重要アウトプット。以下のルールに従う:

### 仮説の構成要素
- **仮説文**: 「〜である」「〜の可能性が高い」の断定形で書く
- **確認ポイント**: 商談で何を聞けばこの仮説が検証できるかを明記
- **根拠**: レポートのどのセクションの情報に基づく仮説かを `>` 引用ブロックで補足

### 仮説のカテゴリ
| カテゴリ | 内容 | 目安数 |
|---------|------|--------|
| **A. 業務課題** | 顧客の現場で起きている課題の仮説 | 5〜8個 |
| **B. 技術ソリューション** | 燈が提案する技術的アプローチの仮説 | 4〜6個 |
| **C. ビジネス** | 市場性・ROI・競合優位性の仮説 | 3〜5個 |
| **D. 深掘り質問** | 仮説を検証するための具体的な質問 | 8〜12個 |

### 仮説の品質基準
- **顧客の言葉に紐付いていること**: minutes/ や emails/ の顧客発言に基づく仮説を優先
- **数字で裏付けられていること**: 業界データ・ベンチマーク等の定量情報を根拠にする
- **反証可能であること**: 商談で「YES/NO」で確認できる粒度にする
- **actionableであること**: 仮説が正しかった場合、提案にどう反映するかが明確

## 7. レポート全体の品質基準

### 最重要: ドメイン知識の深さ
- レポートを読んだ後、顧客と **業務の具体的な話** ができるか？が最大の判定基準
- 「〜って、こういう仕組みですよね？」と自分から切り出せるレベルまで書く
- 専門用語を **正しく使える** レベルで解説する（誤用は信頼を失う）
- 業務の **具体例・ケーススタディ** を含める（抽象論だけでは理解が浅い）

### 形式面
- **> 引用ブロック** を各セクションの最後に配置し、「商談での仮説」として提案への示唆を記載する
- **テーブル（表）** を多用して情報を構造化する
- **数字を必ず含める**: 業務量、処理時間、コスト、件数等。数字のない仮説は弱い
- **出典URL** を参考リンク集にすべて記載する

### アンチパターン（やってはいけないこと）
- 業界トレンドや政策の羅列だけで業務の中身が薄いレポート
- PJの進捗報告とドメイン知識調査を混同する
- 競合サービスの一覧表を並べただけで「何が課題で何が解決できていないか」がないレポート
- 顧客の業務を理解せずに「AI活用で効率化」と書く汎用レポート

## 8. 注意事項

- 案件フォルダ（README, minutes, emails）を **すべて読んでから** Web検索を開始する。顧客コンテキストなしの汎用調査は禁止
- 調査テーマが広すぎる場合、ユーザーに確認してスコープを絞る
- 金額・技術選定に関する仮説は、確度が低い場合は「要確認」と明記する
- レポート生成後、ユーザーに内容を確認してもらう
- 既存の調査レポート（docs/ 内）がある場合は、重複を避けて差分情報を追記する方向で検討する
- Web検索は日本語・英語の両方で行い、国内外の情報を網羅する
